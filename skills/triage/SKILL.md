# Triage

Move issues through a small state machine. Every AI-generated comment on issues must start with:

```
> *Generated by AI during triage.*
```

## Roles

**Category** (exactly one):
- `bug` ÔÇö something is broken
- `enhancement` ÔÇö new feature or improvement

**State** (exactly one):
- `needs-triage` ÔÇö needs evaluation
- `needs-info` ÔÇö waiting on reporter
- `ready-for-agent` ÔÇö fully specified, AFK-ready
- `ready-for-human` ÔÇö needs human judgment
- `wontfix` ÔÇö will not be actioned

## State machine

```
unlabeled Ôćĺ needs-triage Ôćĺ needs-info Ôćĺ needs-triage (when reporter replies)
                         Ôćĺ ready-for-agent
                         Ôćĺ ready-for-human
                         Ôćĺ wontfix
```

Maintainer can override any transition. Flag unusual transitions and ask.

## Invocation

Interpret natural language requests:
- "Show me what needs attention" Ôćĺ present 3 buckets
- "Let's look at #42" Ôćĺ triage specific issue
- "Move #42 to ready-for-agent" Ôćĺ quick override
- "What's ready for agents?" Ôćĺ filter ready-for-agent

## Show what needs attention

Query issues and present 3 buckets (oldest first):

1. **Unlabeled** ÔÇö never triaged
2. **needs-triage** ÔÇö evaluation in progress
3. **needs-info with new reporter activity** ÔÇö needs re-evaluation

Show count + one-line summary per issue. Let user pick.

## Triage a specific issue

1. **Gather context.** Read full issue (body, comments, labels, dates). Check prior triage notes. Explore relevant codebase.
2. **Recommend.** Category + state + reasoning + codebase summary. Wait for direction.
3. **Reproduce (bugs only).** Read reporter's steps, trace code, run tests. Report: confirmed repro / failed repro / insufficient detail (Ôćĺ needs-info signal).
4. **Grill if needed.** Run grill-me to flesh out underspecified issues.
5. **Apply outcome:**
   - `ready-for-agent` Ôćĺ post agent brief (see below)
   - `ready-for-human` Ôćĺ same brief + note why it can't be delegated
   - `needs-info` Ôćĺ post triage notes (see template)
   - `wontfix` (bug) Ôćĺ polite explanation, close
   - `wontfix` (enhancement) Ôćĺ document reasoning, close
   - `needs-triage` Ôćĺ apply label, optional partial-progress comment

## Agent brief format

Durable spec ÔÇö no file paths, no line numbers, no implementation details:

```markdown
## Agent Brief

**Goal:** [what to achieve]

**Behavior:**
- [observable behavior 1]
- [observable behavior 2]

**Acceptance criteria:**
- [ ] [testable criterion 1]
- [ ] [testable criterion 2]

**Context:**
- [relevant domain knowledge]
- [known constraints]
```

## Needs-info template

```markdown
## Triage Notes

**Established:**
- [resolved point 1]
- [resolved point 2]

**Needed from @reporter:**
- [specific actionable question 1]
- [specific actionable question 2]
```

## Quick state override

User says "move #X to Y" Ôćĺ trust them, confirm what you're about to do, then act. Skip grilling. If moving to ready-for-agent without grilling, ask if they want an agent brief.

## Resuming sessions

If prior triage notes exist, read them, check if reporter answered outstanding questions, present updated picture. Don't re-ask resolved questions.

## Hermes-specific adaptations

- Use `terminal` with `gh` CLI for GitHub issue operations
- Use `search_files` and `read_file` for codebase exploration
- Use `delegate_task` for parallel codebase exploration when triaging multiple issues
- Ready-for-agent issues can be fed directly to kanban system or cron jobs
